# 开题报告补充材料

## 一句话摘要

Helios 关注的核心问题是：业务经验如何被持续沉淀成能力。它把自然语言交互、业务知识网络、Agent 编排、权限治理和证据追溯放进同一个闭环，让业务侧不懂研发也能把自己的需求做成可复用的智能流程。

## 命题前置分析与洞察

企业数字化已经积累了大量系统：CRM、项目管理、合同、会议纪要、知识库、经营报表、指标口径表。问题在于，这些系统保存了信息，却很少保存「如何判断」。一线员工遇到真实业务问题时，仍然要跨系统搜索、找人确认、翻历史项目、理解指标口径。资深员工离开后，客户关系、项目经验、风险判断和组织规则也容易随人流失。

从外部趋势看，企业 AI 正进入 Agent 阶段。麦肯锡 2025 年 AI 调研指出，许多组织已经开始实验或部署 AI Agent，但多数企业仍停留在局部试点，尚未形成企业级价值；其中一个关键原因，是 AI 没有真正嵌入工作流和组织机制。Gartner 也预测，任务型 AI Agent 会快速进入企业应用。这个趋势说明，下一阶段的竞争点正在从「有没有大模型入口」转向「能否把数据、规则、权限和流程组织成可持续运营的智能能力」。

对四维图新这类数据密集、项目复杂、业务链条长的企业来说，数据治理的价值不应只停留在数据标准化和报表可视化。更关键的是，让业务人员能用自然语言把需求表达出来，让 Agent 调用被授权的数据和工具，把一次性的查询、判断、复盘沉淀成可复用模板。这样，企业数据才会从「被存储」走向「被调用」，业务经验才会从「靠人传」走向「可继承」。

## 整体解决方案设计

Helios 定位为企业级业务能力沉淀 Agent。它面向客户洞察、项目复盘、合同风险、经营指标、知识查询等高频场景，让业务人员通过自然语言创建、调用和迭代 Agent。系统不要求业务侧理解研发流程，也不要求他们把需求写成复杂产品文档；业务人员只需要描述目标、输入限制和期望输出，Helios 就能将其拆解为可执行任务流，并在权限允许范围内调度数据源、文档、规则和工具。

Helios 的架构由六层能力组成。第一层是意图理解与任务路由，识别用户真正想完成的业务动作，并把自然语言拆成结构化任务。第二层是业务知识网络，将 CRM、项目库、合同、会议纪要、指标口径、组织权限、历史决策和人工修正连接起来，形成企业自己的语义地图。第三层是 Agent 编排引擎，业务人员可以用自然语言定义任务流，例如「查询客户历史合作、对比相似项目、识别风险点、生成拜访建议」，系统自动调度查数、读文档、比口径、找案例、判风险等能力节点。第四层是业务侧权限与上下文治理，每个 Agent 的数据访问范围、可执行动作、审批边界都绑定具体业务角色，能力可以开放，数据不会失控。第五层是证据账本，关键结论都附带来源、版本、时间、口径、置信度和推理路径。第六层是组织记忆回流，高质量问答、人工确认、异常修正和项目复盘会沉淀进知识网络和 Agent 模板。

核心创新在于，Helios 把 Agent 从「研发交付的功能」升级为「业务侧可运营的能力」。传统系统建设通常是业务提需求、产品写方案、研发排期、上线验收，周期长且难以覆盖大量细碎场景。Helios 让业务人员通过自然语言把需求变成 Agent，把一次性的经验判断变成可复用流程，把分散的个人经验变成组织资产。RAG、知识图谱、工作流编排、权限治理和证据追溯共同服务于一个目标：建立企业知识的生产、调用、校准和沉淀机制。

落地可从客户洞察和项目复盘两个场景切入。客户洞察 Agent 帮销售快速理解客户历史合作、关键联系人、合同节点和潜在风险；项目复盘 Agent 帮交付团队从历史文档和会议纪要中提炼经验、风险模式和可复用方案。试点期可以衡量四类指标：重复问询量下降、资料检索时间下降、关键结论证据链覆盖率提升、可复用 Agent 模板数量增长。随着使用增加，Helios 会逐步沉淀出面向销售、交付、经营分析、合同风控和知识管理的 Agent 资产库，成为企业内部持续生长的智能能力层。
